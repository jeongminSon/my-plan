## 1. 비밀번호 찾기 미구현 — Must

**문제**: `LoginScreen.tsx:153`이 여전히 `비밀번호 찾기 (준비 중)`입니다. 이메일+비밀번호로 가입한 사용자가 비번을 잊으면 **계정 복구 경로가 전혀 없습니다.** PRD에서 Fast-follow로 분류했던 항목이며, 실사용 시 이탈·문의로 직결됩니다.

```
LoginScreen의 "비밀번호 찾기 (준비 중)"를 실제 기능으로 구현해줘.
- supabase.auth.resetPasswordForEmail(email, { redirectTo }) 로 재설정 메일 발송
- 재설정 요청 화면(이메일 입력) + 발송 완료 안내 + 메일의 링크로 돌아왔을 때
  새 비밀번호 설정 화면(updateUser({ password })) 처리
- 보안: 존재하지 않는 이메일도 동일하게 "메일을 보냈다"고 안내(이메일 열거 방지)
- 접근성(label, 포커스 링, Enter 제출) 준수
구현 후 검증 함수 단위 테스트를 추가하고 jest/tsc 통과를 확인해줘.
```

## 2. 데이터 유실 방지 로직 테스트 부재 — Should

**문제**: `localCleanup.ts`는 "마이그레이션 완료 플래그가 있을 때만 로컬 캐시를 지운다"는 **유실 방지 안전 가드**를 담고 있습니다. 로직은 올바르지만 **단위 테스트가 없습니다.** 마이그레이션 가드는 데이터 안전의 핵심이라 회귀 보호가 필요합니다.

```
src/auth/localCleanup.ts의 clearLocalCacheOnLogout에 단위 테스트를 추가해줘.
의존성(AsyncStorage, localStorage)을 목으로 주입하거나 모킹해서 다음 케이스를 검증:
1) migrated 플래그 없음 → 아무것도 지우지 않음(미이전 데이터 보존)
2) migrated 플래그 '1' → my-plan.tasks/lists 및 백업 제거
3) userId 없음 → no-op
4) AsyncStorage 오류 → 안전하게 보존 쪽으로 동작
기존 테스트 스타일(순수/주입)에 맞춰 작성하고 전체 테스트 통과를 확인해줘.
```

## 4. 배포·설정 확인 (코드 아님) — Must(확인)

**문제**: 두 가지는 코드로 검증할 수 없어 **대시보드 확인**이 필요합니다.
- `delete-account` Edge Function이 실제로 **배포**됐는지 (미배포 시 데이터는 지워지지만 인증 계정·이메일이 잔존).
- Supabase **"Confirm email"** 설정이 켜져 있는지 (꺼져 있으면 미인증 계정이 그대로 사용 가능).

```
(직접 확인) Supabase 대시보드에서:
1) Edge Functions → delete-account 가 배포돼 있는지 확인. 없으면 `supabase functions deploy delete-account` 실행.
2) Authentication → Providers/Email → "Confirm email"이 켜져 있는지 확인.
두 설정 상태를 알려주면, 미흡한 부분에 대한 후속 조치를 안내할게.
```
